# Tier 3: Advanced Machine Learning 모델별 세부 설계 및 실행 전략

본 문서는 `Tier3_Design.md`의 "2.1 최신 트리 기반 앙상블 알고리즘" 섹션을 구체화하여, 로컬 PC 환경(Intel Core i7-12700, 12코어 20스레드, 32GB RAM)에 맞춘 현실적이고 안정적인 모델 학습 가이드라인을 제공합니다. XGBoost, LightGBM, CatBoost 각각의 특성을 고려한 전처리, 하이퍼파라미터 튜닝 및 OOM(Out-Of-Memory) 방지 전략을 포함합니다.

---

## 1. 하드웨어 및 메모리 제약 고려사항 (Local Environment)

현재 로컬 PC는 32GB의 RAM과 20스레드를 보유하고 있습니다. 최대 데이터셋인 GLAM(약 2,600만 행 × 38 피처)은 `Float64` 기준으로 약 8GB를 차지합니다. 단일 데이터 로드 시에는 문제가 없으나, 교차 검증(CV) 및 데이터 복사 과정에서 OOM(Out-Of-Memory)이 발생할 위험이 높습니다.

### 1.1 전역 메모리 최적화 및 OOM 방지 전략
* **데이터 타입 다운캐스팅:** 모델에 데이터를 주입하기 전, 연속형 피처를 `float64`에서 `float32`로 일괄 다운캐스팅하여 메모리 사용량을 절반 수준으로 줄입니다.
* **명시적 가비지 컬렉션 (Walk-Forward CV):** 시계열 교차 검증은 폴드가 진행될수록 훈련 데이터의 크기가 증가합니다. 각 폴드 학습 및 평가 종료 직후, 불필요해진 훈련/검증 객체(`X_train`, `y_train` 등)를 `del` 키워드로 삭제하고, `gc.collect()`를 명시적으로 호출하여 메모리를 즉각 해제합니다.
* **멀티코어 활용:** CPU 자원(20 논리 스레드)을 적극 활용하기 위해 각 모델별로 스레드/코어 관련 파라미터(`n_jobs=-1`, `thread_count` 등)를 최대로 할당하여 학습 시간을 단축합니다.

---

## 2. 모델별 데이터 전처리 및 입력 객체 최적화

세 가지 트리 기반 앙상블 모델은 내부적인 데이터 처리(결측치, 범주형 변수, 메모리 캐싱) 방식이 다릅니다. 따라서 파이프라인에서 각 모델에 맞는 고유 객체 생성과 데이터 처리가 이루어져야 합니다.

### 2.1 XGBoost (eXtreme Gradient Boosting)
* **입력 데이터 객체:** `xgboost.DMatrix`를 사용하여 메모리 효율을 극대화하고 학습 속도를 높입니다.
* **범주형 변수 처리:** 버전에 따라 `enable_categorical=True` 파라미터를 활성화하여 Pandas의 `category` 타입을 바로 지원하도록 설정하거나, 필요한 경우 사전에 타겟 인코딩(Target Encoding) 또는 원-핫 인코딩(One-Hot Encoding)을 적용하여 파이프라인에 주입합니다.
* **결측치 처리:** `DMatrix` 생성 시 `missing=np.nan` 파라미터를 통해 결측치를 명시적으로 알려주어 XGBoost 내부의 스플릿 방향 학습 알고리즘이 결측치를 최적으로 처리하도록 합니다.

### 2.2 LightGBM
* **입력 데이터 객체:** `lightgbm.Dataset`을 사용합니다. LightGBM의 내부 Dataset은 히스토그램 기반의 최적화된 바이닝(Binning) 정보를 저장하므로 필수적입니다.
* **범주형 변수 처리:** 별도의 인코딩 없이 Pandas DataFrame의 범주형 변수 컬럼을 `category` 데이터 타입으로 변환하여 주입합니다. LightGBM 자체적으로 범주형 특성의 스플릿을 최적화합니다.
* **메모리 관리:** `Dataset` 생성 시 `free_raw_data=True` (기본값) 속성을 활용하여 원본 데이터 복사본을 해제함으로써 메모리를 절약합니다.

### 2.3 CatBoost
* **입력 데이터 객체:** `catboost.Pool` 객체를 활용합니다. 이는 CatBoost의 최적화된 데이터 구조로, 대규모 데이터 처리에 유리합니다.
* **범주형 변수 처리:** 데이터를 숫자나 원-핫 인코딩으로 변환하지 않고 **원본 범주형 변수(문자열 등)를 그대로 유지**합니다. `Pool` 객체 생성 및 학습 시 `cat_features` 파라미터에 범주형 컬럼 인덱스나 이름을 전달하면 내부에서 정교한 범주형 인코딩(Ordered Target Statistics 등)이 수행됩니다.
* **학습 속도 고려:** CatBoost는 범주형 처리에 강점이 있으나 연산량이 많으므로, GPU가 없는 환경에서는 다른 모델 대비 학습 시간이 오래 걸릴 수 있음을 감안합니다.

## 3. Optuna 기반 하이퍼파라미터 튜닝 전략

밤샘(Overnight) 7~8시간 내외의 학습 제약을 고려하여, 데이터셋 크기에 따라 튜닝 자원을 유연하게 분배합니다.

* **동적 `n_trials` 조정:**
  * **초대형 데이터셋 (GLAM, IOBP2 등 1,000만 윈도우 이상):** 탐색 횟수를 10~15회 내외로 타협하여 학습 시간 내에 여러 파라미터 조합을 얕게라도 탐색합니다.
  * **소형 데이터셋 (수십만 윈도우 이하):** 50회 이상의 충분한 `n_trials`를 부여하여 모델의 최대 성능을 끌어냅니다.
* **안전장치 (Timeout 설정):**
  * 밤샘 학습 시 예상치 못한 무한 루프나 특정 폴드에서의 병목 현상을 방지하기 위해, Optuna `study.optimize()` 호출 시 `timeout` 파라터를 반드시 명시합니다 (예: 10시간 = `timeout=36000`). 이를 통해 자원 독점 및 학습 미완료 리스크를 완벽히 통제합니다.
* **Pruning (조기 종료):**
  * Optuna 내부의 Pruner(예: `MedianPruner`, `HyperbandPruner`)를 적극 활용하여, 가망이 없는 하이퍼파라미터 조합은 조기 종료시켜 자원을 아낍니다.
